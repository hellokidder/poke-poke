# 可靠性 TODO

## 事故记录

- **发生时间**：2026-04-13
- **事件描述**：家里电脑此前开发的是 Poke Poke 老版本，依赖脚本注入 hooks。更新版本后，本地注入脚本消失，导致 Claude Code (`cc`) 运行也出现问题。
- **风险级别**：高

这不是单纯的“开发机环境丢了一个脚本”，而是一个典型的**集成配置漂移**问题：

- Poke Poke 的可用性依赖用户家目录里的外部文件，例如 `~/.local/bin/poke-hook` 和 `~/.claude/settings.json`
- 这些文件不受项目仓库版本控制保护，机器迁移、升级、清理或手工修改后都可能失效
- 当前实现缺少持续检测、自动修复和显式告警，问题往往在用户真正运行 `cc` 时才暴露

如果 Poke Poke 已经上线，再发生一次同类问题，对用户来说会是严重事故：通知能力失效只是表象，更糟的是可能连 Claude Code 本身的运行都被影响。

## 当前实现暴露出的缺口

- `src-tauri/src/bin/hook.rs`
  - 负责 `--install / --uninstall / --check`
  - 安装目标是 `~/.local/bin/poke-hook`
  - 配置写入是 `~/.claude/settings.json`
- `src-tauri/src/commands.rs`
  - `check_cc_integration()` 只做一次性检查，不会主动恢复
- `src-tauri/src/tray.rs`
  - 托盘菜单只显示“已连接 / 连接 Claude Code”
  - 没有启动时自检、失效告警、降级提示

结论：当前模型仍然把 hook 安装视为“一次性动作”，而不是一个需要持续维持的系统状态。

## 建议的验收标准

- 删除 `~/.local/bin/poke-hook` 后，Poke Poke 在下次启动时能在 10 秒内发现并提示修复
- 覆盖 `~/.claude/settings.json` 后，Poke Poke 能识别为配置漂移，而不是误报“已连接”
- 执行一键修复后，`cc` 可以继续正常运行，且不破坏用户其他 hook 配置
- 重复执行修复不会产生重复 hook 或脏配置
